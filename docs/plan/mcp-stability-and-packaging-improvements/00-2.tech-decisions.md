## ADR-001: MCP stdio 서버 esbuild 번들링

### 상태: 제안

### 맥락
- `mcp-server.mjs`는 Electron 외부에서 독립 Node.js 프로세스로 실행됨
- 현재 `asarUnpack: node_modules/**`로 전체 node_modules를 이중 추출하여 인스톨러 크기 폭증
- `@modelcontextprotocol/sdk` v1.26.0은 express, hono, ajv 등 20+ 전이 의존성 포함
- `zod`는 package.json에 직접 선언되지 않아 호이스팅 불안정

### 결정
`esbuild`로 `mcp-server.mjs`를 단일 CJS 번들(`mcp-server.bundle.cjs`)로 빌드한다.

### 근거
| 선택지 | 장점 | 단점 |
|--------|------|------|
| **A. esbuild 번들링** (채택) | 단일 파일 ~500KB, node_modules unpack 불필요, 의존성 문제 근본 해결 | 빌드 스텝 추가, sourcemap 디버깅 |
| B. 선택적 asarUnpack | 기존 구조 유지 | 전이 의존성 추적 어려움, SDK 업데이트마다 목록 변경 필요 |
| C. node_modules 전체 unpack 유지 | 변경 없음 | 인스톨러 크기 수백MB 증가, 이중 저장 |

### 구현 방향
```bash
# 빌드 스크립트 추가
npx esbuild src/main/mcp-server.mjs \
  --bundle --platform=node --target=node20 \
  --format=cjs --outfile=src/main/mcp-server.bundle.cjs \
  --external:electron
```

- `package.json` scripts에 `prebuild` 스크립트로 번들링 자동화
- `electron-builder.yml`의 `asarUnpack`에서 `node_modules/**` 제거, `mcp-server.bundle.cjs`만 유지
- `frontmatter.js`도 번들에 인라인되므로 별도 unpack 불필요
- 개발 환경에서는 원본 `.mjs` 직접 실행 유지 (`npm run mcp`)

### 영향
- 인스톨러 크기: 수백MB 감소
- 빌드 파이프라인: `esbuild` devDependency 추가, prebuild 스크립트 추가
- README: 패키징된 앱 경로에서 `.bundle.cjs` 확장자로 변경

## ADR-002: 검색 엔진 동시성 — AsyncMutex 패턴

### 상태: 제안

### 맥락
- `ensureFresh()`가 동시 호출되면 `rebuild()`가 중복 실행
- `rebuild()`는 `this.engine`을 초기화 후 재구축하므로 중간 상태에서 검색 시 빈 결과 반환
- Node.js는 단일 스레드이지만 `await` 포인트에서 다른 비동기 작업이 끼어들 수 있음

### 결정
Promise 기반 간이 mutex를 `ensureFresh()`에 적용한다.

### 근거
| 선택지 | 장점 | 단점 |
|--------|------|------|
| **A. Promise mutex** (채택) | 외부 의존성 없음, Node.js 이벤트 루프 모델에 적합 | 직접 구현 필요 |
| B. `async-mutex` npm 패키지 | 검증된 구현 | 새 의존성 추가 |
| C. dirty flag 개선만 | 최소 변경 | 경쟁 조건 근본 해결 안 됨 |

### 구현 방향
```javascript
// search-engine.js
constructor(store) {
  // ...
  this._rebuildPromise = null;  // 현재 진행 중인 rebuild Promise
}

async ensureFresh() {
  if (!this._initialized || this.dirty) {
    if (this._rebuildPromise) {
      return this._rebuildPromise;  // 이미 진행 중이면 같은 Promise 공유
    }
    this._rebuildPromise = this.rebuild().finally(() => {
      this._rebuildPromise = null;
    });
    return this._rebuildPromise;
  }
}
```

### 영향
- `search-engine.js` 수정: 3개 속성 추가, `ensureFresh()` 로직 변경
- 기존 API 호환: 외부 인터페이스 변경 없음

## ADR-003: HTTP MCP Origin 기반 접근 제어

### 상태: 제안

### 맥락
- 현재 `Access-Control-Allow-Origin: *`로 모든 origin 허용
- 127.0.0.1 바인딩만으로는 DNS rebinding 공격에 취약
- MCP 클라이언트는 보통 CLI/데스크탑 앱이므로 CORS 제한이 기능에 영향 없음

### 결정
CORS 헤더를 제거하고, `Origin` 헤더가 있는 브라우저 요청을 차단한다.

### 근거
| 선택지 | 장점 | 단점 |
|--------|------|------|
| **A. Origin 헤더 차단** (채택) | 브라우저 CSRF/rebinding 차단, 기존 CLI 클라이언트 영향 없음 | curl에 Origin 헤더 붙이면 차단됨 |
| B. 토큰 인증 | 강력한 보안 | 설정 복잡성 증가, UX 저하 |
| C. allowlist 기반 CORS | 유연함 | 허용 origin 목록 관리 필요 |

### 구현 방향
```javascript
// mcp-http.mjs — 요청 핸들러 시작부
const origin = req.headers['origin'];
if (origin) {
  res.writeHead(403, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify({ error: 'Browser requests not allowed' }));
  return;
}
// CORS 헤더 완전 제거
```

### 영향
- 브라우저 기반 MCP 클라이언트는 사용 불가 (현재 해당 없음)
- curl, Claude Desktop, Claude Code 등 CLI/네이티브 클라이언트 정상 작동
- `OPTIONS` preflight 핸들러 제거 가능

## ADR-004: saveMcpFile 공유 모듈 추출

### 상태: 제안

### 맥락
- `saveMcpFile()` 함수가 `index.js` (lines 611-660)와 `mcp-http.mjs` (lines 46-95)에 거의 동일하게 중복
- `sanitizeFilenameWithUrlEncode()`와 `extractTitleFromContent()`도 양쪽에 중복
- 버그 수정이나 기능 변경 시 양쪽 동기화를 잊기 쉬움

### 결정
`src/main/mcp-save.js` (CJS)로 공통 함수를 추출한다.

### 근거
- CJS로 작성하면 `index.js`(CJS)에서 `require()`, `mcp-http.mjs`(ESM)에서 `createRequire()`로 사용 가능
- `mcp-server.mjs`(ESM, 독립 프로세스)에서도 필요 시 `createRequire()`로 사용 가능
- esbuild 번들링 시 mcp-server.bundle.cjs에 인라인됨

### 영향
- 새 파일 1개 생성: `src/main/mcp-save.js`
- `index.js`와 `mcp-http.mjs`에서 해당 함수 제거 후 import로 교체

## ADR-005: Named Window 동시성 — Pending Lock Map

### 상태: 제안

### 맥락
- 동일 `windowName`으로 동시 `open_markdown` 요청 시 2개 창 생성 가능
- `getWindowByName()` 조회와 `nameToId.set()` 등록 사이에 간격 존재
- Node.js 단일 스레드이지만 `createWindow()`가 async이므로 await 사이에 끼어들기 가능

### 결정
`nameToId`와 별도로 `_pendingNames` Set을 도입하여 이름 잠금을 즉시 확보한다.

### 구현 방향
```javascript
// window-manager.js
async createWindow(opts) {
  if (opts.windowName) {
    if (this._pendingNames.has(opts.windowName)) {
      // 이미 생성 중이면 짧은 대기 후 upsert 시도
      await this._waitForPendingName(opts.windowName);
      return this.createWindow(opts);  // 재귀 (이제 기존 창 발견됨)
    }
    this._pendingNames.add(opts.windowName);  // 즉시 잠금
  }
  try {
    // ... 기존 로직
  } finally {
    if (opts.windowName) this._pendingNames.delete(opts.windowName);
  }
}
```

### 영향
- `window-manager.js` 수정: `_pendingNames` Set 추가, `createWindow()` 로직 래핑

## ADR-006: 인덱스 손상 복구 — dirty 자동 설정

### 상태: 제안

### 맥락
- 인덱스 파일 손상(부분 기록, JSON 파싱 실패) 시 빈 엔진으로 시작
- `dirty = false`로 남아 있어 `ensureFresh()`가 재빌드를 트리거하지 않음
- 수동 `markDirty()` 이벤트(새 문서 저장)가 있을 때까지 검색 기능 비활성

### 결정
`_loadIndex()` 실패 시 `this.dirty = true`를 설정하고, `initialize()` 후 즉시 비동기 재빌드를 시도한다.

### 영향
- `search-engine.js`의 `initialize()` catch 블록 수정 (1줄 추가)

## ADR-007: update_markdown 자동 저장 추가

### 상태: 제안

### 맥락
- `open_markdown`만 `saveMcpFile()` 호출
- `update_markdown`으로 내용이 변경되어도 디스크에 저장되지 않음
- 검색 인덱스가 최초 내용만 반영하여 stale 상태

### 결정
`update_markdown` IPC 핸들러에서도 `saveMcpFile()` 호출 및 `searchEngine.markDirty()` 실행.

### 구현 방향
- `index.js`의 `update_markdown` 핸들러에 `saveMcpFile()` 호출 추가
- `mcp-http.mjs`의 `update_markdown` tool 핸들러에도 동일 적용
- `savedFilePath`가 이미 있으면 같은 경로에 덮어쓰기, 없으면 새 파일 생성

### 영향
- `index.js` 수정: ~15줄 추가
- `mcp-http.mjs` 수정: ~15줄 추가 (공유 모듈 추출 후에는 호출 1줄)
