## 구현 체크리스트

### FR-4-001: Named Window _pendingNames 잠금 도입

**파일**: `src/main/window-manager.js`
**변경 위치**: constructor (line 76-104), createWindow() (line 264-460)

```javascript
// constructor에 추가 (line ~103)
this._pendingNames = new Set();

// createWindow() 시작부 수정 (line ~271)
async createWindow(opts) {
  const windowName = opts.windowName;

  if (windowName) {
    // 1. 이미 존재하는 named window → upsert
    const existingId = this.getWindowByName(windowName);
    if (existingId != null) {
      return this.updateWindow(existingId, opts);
    }

    // 2. 생성 진행 중인 동일 이름 → 대기 후 재시도 (최대 5초)
    if (this._pendingNames.has(windowName)) {
      const PENDING_TIMEOUT = 5000;
      const start = Date.now();
      await new Promise((resolve, reject) => {
        const check = () => {
          if (!this._pendingNames.has(windowName)) return resolve();
          if (Date.now() - start > PENDING_TIMEOUT) {
            return reject(new Error(`Timed out waiting for windowName "${windowName}"`));
          }
          setTimeout(check, 50);
        };
        setTimeout(check, 50);
      });
      return this.createWindow(opts);  // 재귀: 이제 기존 창 발견됨
    }

    // 3. 이름 잠금 확보
    this._pendingNames.add(windowName);
  }

  try {
    // ... 기존 BrowserWindow 생성 로직 ...
    // nameToId.set()은 기존 위치 유지 (line ~391)
  } finally {
    if (windowName) this._pendingNames.delete(windowName);
  }
}
```

- [ ] `_pendingNames` Set을 constructor에 추가
- [ ] `createWindow()` 시작부에 이름 잠금 체크 로직 추가
- [ ] 잠금 해제를 `finally` 블록에서 보장
- [ ] 대기 중 재시도 시 기존 창을 발견하여 upsert로 전환
- [ ] 최대 대기 시간 제한 (타임아웃, 예: 5초)

### FR-4-003: IPC 요청 1회 재시도 로직

**파일**: `src/main/mcp-server.mjs`
**변경 위치**: sendIpcRequest() (line 217-245)

```javascript
async function sendIpcRequest(action, params, _retried = false) {
  if (!ipcSocket || ipcSocket.destroyed) {
    await connectToElectron();
  }

  return new Promise((resolve, reject) => {
    const id = randomUUID();
    const timer = setTimeout(() => {
      pendingRequests.delete(id);
      reject(new Error(`IPC request timed out (${IPC_TIMEOUT}ms)`));
    }, IPC_TIMEOUT);

    pendingRequests.set(id, { resolve, reject, timer });

    try {
      const payload = JSON.stringify({ id, action, params }) + '\n';
      const ok = ipcSocket.write(payload);
      if (!ok && !_retried) {
        // 버퍼 풀: 한 번 재시도
        pendingRequests.delete(id);
        clearTimeout(timer);
        ipcSocket = null;
        return sendIpcRequest(action, params, true).then(resolve, reject);
      }
    } catch (err) {
      pendingRequests.delete(id);
      clearTimeout(timer);
      if (!_retried) {
        ipcSocket = null;
        return sendIpcRequest(action, params, true).then(resolve, reject);
      }
      reject(err);
    }
  });
}
```

- [ ] `_retried` 매개변수로 최대 1회 재시도 제한
- [ ] `socket.write()` 반환값 `false` 체크 (버퍼 풀 상태)
- [ ] write 실패 시 소켓 초기화 후 재연결+재시도
- [ ] 재시도에서도 실패 시 원래 에러 전파

## 테스트 시나리오

### TC-4-001: 동일 windowName 동시 요청 (정상 케이스)

**Given**: windowName "status" 창이 존재하지 않음
**When**: 2개의 `open_markdown(windowName: "status")` 요청이 동시에 도착
**Then**:
- 창이 정확히 1개만 생성됨
- 두 번째 요청은 첫 번째 창의 upsert로 처리됨
- `nameToId` Map에 "status" 엔트리가 1개만 존재

### TC-4-002: Named Window 생성 후 즉시 업데이트 (정상 케이스)

**Given**: windowName "log" 창이 존재하지 않음
**When**: `open_markdown(windowName: "log")` 직후 `open_markdown(windowName: "log", content: "updated")` 호출
**Then**:
- 첫 번째 요청으로 창 생성
- 두 번째 요청으로 내용 업데이트 (upsert)
- 최종 내용이 "updated"

### TC-4-003: IPC 소켓 끊김 후 자동 재연결 (예외 케이스)

**Given**: MCP stdio 서버가 Electron과 IPC 연결된 상태
**When**: Electron 앱이 재시작됨 (IPC 소켓 끊김)
**Then**:
- 다음 MCP tool 호출 시 자동 재연결 시도
- 재연결 성공 후 tool 호출 정상 처리
- 에러 없이 결과 반환

### TC-4-004: 동시 연결 시도 통합 (예외 케이스)

**Given**: IPC 소켓이 끊긴 상태
**When**: 3개의 MCP tool 호출이 동시에 `connectToElectron()` 호출
**Then**:
- 실제 연결 시도는 1회만 수행
- 3개 모두 같은 연결 완료를 기다림
- 연결 후 3개 요청 모두 정상 처리

### TC-4-005: _pendingNames 타임아웃 (경계값)

**Given**: windowName "test" 잠금 획득 후 창 생성 중 에러 발생
**When**: 두 번째 `open_markdown(windowName: "test")` 요청이 대기 중
**Then**:
- `finally` 블록에서 잠금 해제
- 두 번째 요청이 정상 진행 (새 창 생성)
- 잠금이 영구적으로 남지 않음
