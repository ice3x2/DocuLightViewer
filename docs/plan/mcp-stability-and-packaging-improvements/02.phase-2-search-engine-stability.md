### FR-2-001: ensureFresh() Promise mutex 도입

**파일**: `src/main/search-engine.js`
**변경 위치**: constructor (line 15-21), ensureFresh() (line 183-187)

```javascript
// constructor에 추가
this._rebuildPromise = null;

// ensureFresh() 수정
async ensureFresh() {
  if (!this._initialized || this.dirty) {
    if (this._rebuildPromise) {
      return this._rebuildPromise;
    }
    this._rebuildPromise = this.rebuild().finally(() => {
      this._rebuildPromise = null;
    });
    return this._rebuildPromise;
  }
}
```

- [ ] `_rebuildPromise` 속성 추가
- [ ] 동시 호출 시 동일 Promise 반환 확인
- [ ] rebuild 완료 후 `_rebuildPromise = null` 리셋 확인
- [ ] rebuild 중 `markDirty()` 호출 시 다음 ensureFresh에서 재빌드 트리거 확인

// 변경 후: 배치 병렬 읽기
const BATCH_SIZE = 20;
for (let i = 0; i < files.length; i += BATCH_SIZE) {

    batch.map(f => fs.promises.readFile(f, 'utf-8').catch(() => null))

- [ ] `BATCH_SIZE = 20` 상수 정의

## 테스트 시나리오

### TC-2-001: 동시 검색 요청 시 중복 rebuild 방지 (정상 케이스)

**Given**: 검색 인덱스가 dirty 상태
**When**: 2개의 `search_documents` 요청이 동시에 도착
**Then**:
- `rebuild()`가 정확히 1번만 실행됨
- 두 요청 모두 동일한 rebuild 완료 후 결과 반환
- 결과가 일관됨 (빈 결과 아님)

### TC-2-002: rebuild 중 새 문서 저장 (정상 케이스)

**Given**: rebuild()가 진행 중
**When**: `open_markdown`으로 새 문서가 저장되어 `markDirty()` 호출
**Then**:
- 현재 rebuild는 정상 완료
- `dirty` 플래그가 true로 설정됨
- 다음 `ensureFresh()` 호출 시 재빌드 실행

### TC-2-003: 손상된 인덱스 파일 복구 (예외 케이스)

**Given**: 인덱스 파일이 부분 기록으로 JSON 파싱 불가
**When**: 앱 시작 시 `initialize()` 호출
**Then**:
- `_loadIndex()` 실패 로그 출력
- `dirty = true` 설정
- 다음 검색 시 `ensureFresh()` → 자동 `rebuild()` 실행
- 검색 결과 정상 반환

### TC-2-004: update_markdown 후 검색 반영 (예외 케이스)

**Given**: `open_markdown`으로 "Hello World" 문서 열기, 자동 저장됨
**When**: `update_markdown`으로 내용을 "Hello Universe"로 변경
**Then**:
- 변경된 내용이 자동 저장 파일에 반영
- `searchEngine.markDirty()` 호출
- "Universe"로 검색 시 해당 문서 검색됨

### TC-2-005: 깊은 디렉토리 트리 스캔 (경계값)

**Given**: mcpAutoSavePath 아래에 11단계 깊이의 중첩 디렉토리 존재
**When**: `_scanMarkdownFiles()` 실행
**Then**:
- 10단계까지만 스캔
- 11단계 이후 파일은 무시
- 경고 로그 출력
- 스택 오버플로우 없음
